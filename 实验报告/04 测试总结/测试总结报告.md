## 二、被测软件简介

### 2.1 软件概述

"火眼金睛无微不至测量系统"是一款专业的图像测量软件，主要用于测量图像中物体的大小。该系统通过图像处理技术，能够自动或手动标定图像比例尺，检测并测量图像中目标物体的尺寸，并生成详细的测量统计报告。

### 2.2 主要功能模块

系统主要包含以下功能模块：

| 功能模块 | 主要功能描述 |
|----------|------------|
| **系统功能** | 打开图像、删除图像、数据清理 |
| **视图功能** | 检测图像管理（图像显示、缩放调整、清除检测结果等）、识别结果显示、快速访问工具栏、状态栏、菜单栏 |
| **标定功能** | 自动标定、手动标定 |
| **测量功能** | 检测图像（检测整张图内的物体大小）、选取目标（检测选取区域内物体大小） |
| **检测图像** | 显示检测物体图像及直径信息、设置显示图像大小、定位物体位置 |
| **统计分析** | 物体大小级别统计、复制/导出统计结果 |
| **系统设置** | 项目信息管理、测试标准设置 |
| **打印功能** | 打印设置、打印预览 |

### 2.3 技术特点

- 支持多种图像格式（JPG、PNG等）
- 提供自动标定和手动标定两种标定方式
- 自动检测图像中的物体并测量尺寸
- 支持测量结果统计和导出
- 提供打印功能便于报告输出

### 4.2 测试用例设计

本次测试采用多种测试用例设计方法，确保测试的全面性和有效性：

#### 4.2.1 等价类划分法

针对各输入框和功能按钮，划分有效等价类和无效等价类：

| 功能模块 | 输入项 | 有效等价类 | 无效等价类 |

| 手动标定 | 实际距离 | 正数（如100、500.5） | 0、负数、空值、超大数 |
| 测试标准 | 最小/最大直径 | 正整数且最小≤最大 | 最小>最大、空值、负数 |
| 项目信息 | 项目名称 | 1-255个字符 | 空值、256个以上字符 |

| 测试项 | 边界值 | 测试点 |

| 字符串长度 | 1-255 | 0、1、2、254、255、256 |
| 数值输入下限 | ≥1 | -1、0、1、2 |
| 打印预览缩放 | 0.5-5.0 | 0.1、0.5、1.0、2.0、5.0 |

#### 4.2.4 错误推测法

### 6.2 建议

#### 6.2.1 系统存在问题的说明

本次测试揭露了以下主要软件缺陷和不足：

**1. 异常处理机制缺失**
- 软件对异常文件（损坏文件、0字节文件、非图像文件）没有有效的预检查和错误处理
- 建议在文件加载前进行格式验证和完整性检查

**2. 测量算法问题**
- 测量结果存在系统性偏差（约20%）
- 建议检查比例尺计算算法和坐标转换逻辑

**3. 输入验证不完善**
- 多处可通过复制粘贴绕过输入限制
- 边界值检测不完整
- 建议在数据处理层而非仅UI层进行验证

**4. 按钮状态管理混乱**
- 取消按钮在多种条件下失效
- 部分按钮在不应启用时启用
- 建议重新审视按钮状态管理逻辑

#### 6.2.2 可能存在的潜在缺陷和后续工作

**潜在缺陷**：
1. 除已发现的BMP格式问题外，可能还存在其他图像格式的兼容性问题
2. 大尺寸图像处理可能存在性能瓶颈或内存泄漏
3. 数据库操作可能存在并发问题
4. 长时间运行可能存在资源未释放问题

**后续工作建议**：
1. 进行更全面的兼容性测试（更多图像格式、更多操作系统版本）
2. 进行压力测试和长时间运行稳定性测试
3. 缺陷修复后进行回归测试
4. 考虑引入自动化测试以提高测试效率

#### 6.2.3 对缺陷修改和产品设计的建议

**修复优先级建议**：

| 优先级 | 缺陷类型 | 修复时限建议 |
|--------|----------|-------------|
| P0-紧急 | 测量精度问题（CL-MBJC-100） | 立即修复 |
| P0-紧急 | 程序崩溃类错误（R-TX-01/02/03、R-BD-49等） | 立即修复 |
| P1-高 | BMP格式支持问题（F-TX-02） | 1周内修复 |
| P1-高 | 项目信息模块严重错误 | 1周内修复 |
| P2-中 | 一般功能性错误 | 2周内修复 |
| P3-低 | 微小错误 | 下一版本修复 |

**产品设计建议**：

1. **错误处理机制**
   - 建立统一的异常处理框架
   - 对所有外部输入进行验证
   - 提供友好的错误提示信息

2. **UI交互规范**
   - 制定按钮状态管理规范
   - 统一输入框验证规则
   - 规范提示信息格式

3. **代码质量**
   - 建议进行代码审查
   - 增加单元测试覆盖
   - 建立编码规范

#### 6.2.4 对过程改进方面的建议

1. **测试流程改进**
   - 建议在开发阶段就引入测试人员参与
   - 建立缺陷跟踪机制，确保每个缺陷都能被追踪到修复
   - 开发人员修复缺陷时应填写缺陷原因和解决方式

2. **沟通协作改进**
   - 建立开发和测试的定期沟通机制
   - 开发人员发现问题应及时通知测试人员
   - 统一测试环境配置，减少环境差异导致的问题

3. **文档规范改进**
   - 制定统一的编码标准和数据库标准
   - 建立需求变更控制流程
   - 完善使用说明书，明确边界条件和异常处理要求
